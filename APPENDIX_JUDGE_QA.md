# Appendix: 심사위원 예상 질문과 방어 논리

## 1. 가장 먼저 들어올 질문 축

### Q1. 현재 사용자 보유 카드는 어떤 기준으로 골랐나요? 평균 사용자를 대표하나요?

냉정한 질문:

- 이 사용자가 왜 주유 특화 카드, 포인트 카드, 실적 채우기 카드까지 다 가지고 있나요?
- 평균적인 Samsung Wallet 사용자를 대표한다고 보기 어렵지 않나요?
- 데모를 위해 유리한 카드만 넣은 것 아닌가요?

답변 방향:

- 이 데모의 사용자는 `평균 사용자`가 아니라 `혜택 민감형 갤럭시 사용자` 페르소나다.
- 카드 구성은 유리한 카드만 넣은 것이 아니라, 추천 판단에 필요한 대표 케이스를 넣었다.
- 주유 특화 카드, 조건 불일치 카드, 실적 근접 카드, 포인트형 카드를 함께 넣어 추천이 단순 할인액 비교가 아니라는 점을 보여준다.

짧은 답변:

> 평균 사용자를 일반화하려는 카드 구성이 아니라, 결제 직전 판단 비용이 큰 혜택 민감형 사용자를 기준으로 잡았습니다. 일부러 주유 특화 카드, 조건이 맞지 않는 카드, 실적이 가까운 카드, 포인트형 카드를 함께 넣어 SWAP이 어떤 기준으로 후보를 제외하거나 추천하는지 보여주려 했습니다.

장표 포인트:

- `평균 사용자`가 아니라 `핵심 사용 시나리오`
- 추천 로직 검증용 카드 4종
- 할인, 실적, 포인트, 조건 확인을 모두 포함

## 2. 결제 전인데 어떻게 할인액을 아나요?

냉정한 질문:

- 주유소에서는 결제 전 실제 주유 금액을 모르는데, 어떻게 4,500원 혜택을 말하나요?
- 결제 전 예상액을 보여주면 과장 또는 오인 아닌가요?

답변 방향:

- 결제 전에는 원 단위 절감액을 확정하지 않는다.
- 결제 전에는 할인율, 잔여 한도, 선택 정유사 일치 여부, 실적 충족 여부만 보여준다.
- 실제 승인금액 기준 혜택 계산은 결제 후 결과 화면에서만 한다.

짧은 답변:

> 맞습니다. 그래서 결제 전에는 4,500원처럼 원 단위 금액을 확정하지 않고, `SK에너지 10% 할인 가능`, `주유 할인 대상 이용금액 92,000원 남음`처럼 조건 중심으로 보여줍니다. 실제 승인금액이 생긴 뒤에만 결과 화면에서 45,000원 기준 4,500원 할인처럼 계산합니다.

장표 포인트:

- 결제 전: 가능 조건, 잔여 한도, 실적 상태
- 결제 후: 승인금액 기준 계산
- `예상/가능/확인` 톤 유지

## 3. 데이터는 어디서 가져오나요?

냉정한 질문:

- 카드 실적, 잔여 한도, 선택 정유사, 쿠폰 보유 여부를 Samsung Wallet이 어떻게 압니까?
- 카드사별 혜택 조건이 너무 복잡한데, 정말 자동화할 수 있나요?
- 선택형 혜택 카드는 사용자가 카드사 앱에서 직접 고르는데, 그 설정까지 알 수 있나요?

답변 방향:

- 데모는 뱅크샐러드/마이데이터 또는 카드사 제휴 API로 확인 가능한 정보와, 사용자가 Wallet에 등록한 혜택 정보를 전제로 한다.
- 확인 가능한 정보와 확인이 필요한 정보를 UI에서 구분한다.
- 선택 정유사처럼 확정성이 낮은 항목은 `조건 확인`, `카드사 설정 확인`으로 표현한다.

짧은 답변:

> 모든 조건을 Samsung Wallet이 단독으로 안다고 가정하지 않았습니다. 보유 카드, 실적, 한도, 포인트처럼 제휴나 마이데이터로 확인 가능한 정보는 추천에 쓰고, 선택 정유사처럼 확정성이 낮은 정보는 `확인 필요` 상태로 둡니다. 그래서 UI에서도 확정 추천과 조건 확인 후보를 나눠 보여줍니다.

장표 포인트:

- 확인 가능: 보유 카드, 이용내역, 실적, 포인트, 잔여 한도
- 확인 필요: 선택형 혜택, 쿠폰 세부 조건, 청구할인 최종 확정
- 모르면 제외가 아니라 `조건 확인` 후보로 처리

## 4. 삼성페이에 광고처럼 느껴지지 않나요?

냉정한 질문:

- Samsung Pay의 핵심은 빠르고 조용한 결제인데, 추천 배너가 광고처럼 보이면 역효과 아닌가요?
- 카드 추천이 특정 카드사 광고나 제휴 노출처럼 보일 수 있지 않나요?
- 사용자가 원하지 않으면 꺼야 하는데, 그 선택권이 있나요?

답변 방향:

- 추천은 결제 CTA를 대체하지 않고 위에 얹히는 선택 보조 레이어다.
- 사용자는 추천 배너를 접거나, 기존 카드로 바로 결제하거나, 설정에서 끌 수 있어야 한다.
- 정렬 기준은 광고 수수료가 아니라 사용자가 이미 보유한 카드/혜택 기준임을 명시한다.

짧은 답변:

> 그 우려 때문에 추천을 결제 버튼 대신 넣지 않고, 기존 결제 버튼 위의 보조 레이어로 설계했습니다. 사용자는 추천을 접거나 무시하고 기존 카드로 결제할 수 있고, 설정에서 노출 방식을 조절할 수 있습니다. 또한 신규 카드 발급 광고가 아니라 사용자가 이미 보유한 카드의 활용 안내로 제한합니다.

장표 포인트:

- 접기, 바로 결제, 추천 끄기
- 보유 카드 기준 추천
- paid ranking 아님

## 5. 위치/금융 데이터 결합은 민감하지 않나요?

냉정한 질문:

- 위치와 결제 데이터를 묶는 순간 개인정보 부담이 커집니다.
- Samsung이 사용자의 방문 매장을 계속 추적한다는 인상을 줄 수 있습니다.
- 서버로 보내면 규제/신뢰 리스크가 있지 않나요?

답변 방향:

- 상시 추적이 아니라 결제 맥락에서의 온디바이스 후보 인식으로 설명한다.
- 정확한 위치를 단정하지 않고, 사용자가 매장을 확인/수정하게 한다.
- 위치와 금융 데이터는 서버 집적이 아니라 기기 내 처리 및 최소 전송 원칙으로 방어한다.

짧은 답변:

> 이 기능은 사용자를 계속 추적하는 서비스가 아니라, Wallet을 여는 결제 맥락에서 주변 매장 후보를 기기 안에서 확인하는 구조로 설명해야 합니다. 실제 매장이 다를 수 있기 때문에 사용자가 직접 수정할 수 있고, 금융/위치 데이터는 서버에 모아 광고 랭킹을 만드는 방향이 아니라 온디바이스 처리와 최소 활용을 전제로 합니다.

장표 포인트:

- 상시 추적 아님
- 매장 후보 제안 + 사용자 확인
- 온디바이스/최소 활용

## 6. 추천이 틀리면 누가 책임지나요?

냉정한 질문:

- 할인 조건이 틀리면 사용자는 손해를 봅니다.
- 청구할인이나 실적 산입은 카드사 기준인데, Wallet이 보장할 수 있나요?
- 잘못 추천되었을 때 CS 리스크가 큽니다.

답변 방향:

- UI에서 확정/예상/확인 필요 상태를 분리한다.
- 카드사 앱 최종 확인 CTA를 둔다.
- 추천은 금융상품 권유가 아니라 보유 자산 사용 보조 정보로 포지셔닝한다.

짧은 답변:

> 그래서 표현을 `확정 혜택`이 아니라 `예상`, `가능`, `확인 필요`로 나눕니다. 결제 후에도 카드사 앱에서 청구할인, 실적 산입, 잔여 한도를 최종 확인할 수 있게 연결합니다. SWAP의 역할은 혜택을 보장하는 것이 아니라 결제 직전 판단 비용을 줄이는 정보 제공입니다.

장표 포인트:

- 확정/예상/확인 필요 상태 분리
- 카드사 앱 확인
- 보장보다 정보 제공

## 7. 쿠폰/멤버십 사용 여부를 바로 확인할 수 있나요?

냉정한 질문:

- 바코드를 보여줬다고 실제 쿠폰이 사용됐는지 어떻게 압니까?
- 금액권 쿠폰이면 잔액까지 알아야 하지 않나요?
- 멤버십 적립 실패는 어떻게 처리하나요?

답변 방향:

- 데모에서는 `바코드 제시`와 `사용 완료 버튼`으로 사용자 행동을 기록한다.
- 실제 사용/잔액 확정은 제휴 API 또는 카드사/멤버십사 사후 데이터가 필요하다.
- 확인이 어려운 경우 결과 화면에서 `확인 중`, `앱에서 확인` 상태로 둔다.

짧은 답변:

> 바코드를 보여주는 것과 실제 사용 확정은 구분해야 합니다. 데모에서는 사용자가 쿠폰/멤버십을 제시했다는 행동 흐름을 정리하고, 실제 사용 여부나 잔액은 제휴 데이터가 있을 때 확정할 수 있습니다. 없을 경우 결과 화면에서 확인 중 상태나 해당 앱 확인 CTA로 처리합니다.

장표 포인트:

- 제시와 사용 확정 분리
- 금액권 잔액은 제휴 데이터 필요
- 확인 중 상태 제공

## 8. 왜 이게 삼성 MX에 중요한가요?

냉정한 질문:

- 이건 카드사나 핀테크가 할 일 아닌가요?
- Samsung Wallet이 굳이 복잡한 혜택 추천까지 해야 하나요?
- 하드웨어 사업부 관점의 이득이 무엇인가요?

답변 방향:

- 삼성은 OS/디바이스/Wallet 결제 레이어를 갖고 있어 결제 직전 맥락에 가장 자연스럽게 개입할 수 있다.
- 결제 실행만으로는 차별화가 줄어들고 있으므로, 결제 전 의사결정 보조가 Wallet 체류와 Galaxy lock-in을 만든다.
- 금융상품 판매보다 갤럭시 사용 경험 강화로 포지셔닝한다.

짧은 답변:

> 핀테크 앱도 혜택 계산은 할 수 있지만, 결제 직전 사용자가 실제로 Wallet을 여는 순간에 개입하기는 어렵습니다. 삼성은 디바이스와 Wallet 결제 레이어를 갖고 있기 때문에, 결제 실행 직전에 가장 자연스럽게 선택 보조를 제공할 수 있습니다. MX 관점에서는 결제 기능 자체보다 Galaxy에서만 매끄러운 소비 경험을 만드는 것이 핵심입니다.

장표 포인트:

- 결제 직전 맥락
- Wallet 활성화
- Galaxy lock-in

## 9. MVP 범위는 어디까지인가요?

냉정한 질문:

- 지금 설명은 카드, 쿠폰, 위치, 멤버십, 마이데이터, 온디바이스 AI가 다 들어가서 너무 큽니다.
- 실제 MVP로 무엇부터 할 건가요?

답변 방향:

- MVP는 모든 카드/모든 쿠폰 자동화를 목표로 하지 않는다.
- 1차는 등록 카드 + 공개 혜택 조건 + 위치 업종 후보 + 실적/한도 일부 연동으로 제한한다.
- 2차에서 카드사/멤버십 제휴 확장, 3차에서 개인화 기준 고도화로 간다.

짧은 답변:

> MVP는 모든 혜택을 완전 자동화하는 것이 아니라, 등록 카드와 공개 혜택 조건, 위치 업종 후보, 확인 가능한 실적/한도 정보부터 시작하는 구조가 맞습니다. 쿠폰 잔액이나 멤버십 사용 확정처럼 제휴가 필요한 영역은 2단계 이후로 분리해야 현실적입니다.

장표 포인트:

- 1차: 카드 추천/기준 변경/위치 수정
- 2차: 쿠폰/멤버십 제휴
- 3차: 온디바이스 개인화
